If you are looking for a developer-friendly way to make slides, Slide-CN and Slidev will probably end up on the same shortlist.

That makes sense. Both move away from traditional slide software. Both are built for people who are comfortable with code. Both make more sense for technical presentations than PowerPoint-style workflows.

But they are not trying to solve the exact same problem.

## The short version

Use **Slidev** if you want a markdown-first slide authoring experience with strong defaults and a fast path from idea to presentation.

Use **Slide-CN** if you want your slides to behave like part of your React product stack, with reusable components, design-system alignment, and richer UI composition.

That is the real distinction:

- Slidev is optimized for writing slides
- Slide-CN is optimized for building slides

Both approaches are valid. The better choice depends on the kind of presentation you are trying to make.

## Where Slidev is stronger

Slidev has a very appealing workflow for many technical talks.

You can write in markdown, move quickly, and get a polished deck without building a lot of infrastructure around it. For many speakers, that is exactly the right level of abstraction.

Slidev is especially strong when:

- the talk is content-heavy
- markdown is your preferred writing environment
- you want presentation-specific features out of the box
- speed matters more than integrating with an existing component system

If your ideal workflow is "write the story first, lightly customize later," Slidev is a strong fit.

## Where Slide-CN is stronger

Slide-CN is built around a different assumption:

## a presentation can be an extension of your application layer

That means the value shows up when you want to use the same primitives you already use elsewhere:

- React components
- Tailwind tokens
- shadcn-style building blocks
- design-system conventions
- interactive UI states

Instead of treating the deck as a separate document format, Slide-CN treats it more like a small interface with a narrative structure.

That is useful when the presentation is close to the product itself:

- startup pitch decks
- product launches
- developer tool demos
- architecture talks
- presentations that include live UI patterns

## The biggest philosophical difference

Slidev starts from authoring.

Slide-CN starts from composition.

That sounds subtle, but it changes almost everything.

With Slidev, the deck is primarily a presentation document that developers can enhance.

With Slide-CN, the deck is primarily a React surface designed for presentations.

That difference affects how you think about:

- layout
- reuse
- customization
- interactivity
- team workflows

## Example: a branded product presentation

Imagine your team wants a deck that feels exactly like the product marketing site and product UI.

In Slide-CN, that can mean:

- using your real tokens
- sharing the same utility classes
- bringing in UI-inspired components
- rendering interactive fragments in JSX

That is where Slide-CN feels natural.

In Slidev, you can absolutely create polished branded presentations too, but the workflow is not centered on reusing a React design system as the primary building material.

## Example: a technical talk with progressive explanation

Both tools can support code-heavy and technical talks.

The difference is what you want the slides to do.

If you mostly want to write a great technical narrative with code snippets and strong pacing, Slidev may feel faster.

If you want the talk to behave more like an interface, with custom layout composition and React-driven states, Slide-CN becomes more compelling.

## When Slide-CN is the wrong choice

This is the part comparison posts often skip.

You should probably not choose Slide-CN if:

- you want a markdown-first workflow
- you do not want to touch JSX
- the presentation does not benefit from component reuse
- you prefer presentation-specific tooling over app-like flexibility

That does not make Slide-CN worse. It just means the product has a more specific point of view.

## When Slide-CN is the right choice

Slide-CN is a strong fit when:

- your team already uses React
- you care about visual consistency with your product
- you want presentations to reuse system thinking, not imitate it
- interactivity and layout composition matter
- you want to build decks with components instead of documents

## A practical rule of thumb

Ask one question:

## Is this deck closer to a document, or closer to a product surface?

If it is closer to a document, Slidev is likely the better fit.

If it is closer to a product surface, Slide-CN is likely the better fit.

That is the cleanest way to think about the decision.

## Final take

Slidev is not "the old way" and Slide-CN is not "the advanced way."

They represent different philosophies:

- one optimizes for authoring speed and markdown-centric storytelling
- the other optimizes for React-native composition and design-system reuse

For Slide-CN, that focus is deliberate.

The goal is not to replace every developer slide workflow.

The goal is to give React teams a presentation framework that feels like it belongs in the rest of their stack.
