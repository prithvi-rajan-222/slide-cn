---
title: Create an Interactive Technical Talk
description: Technical talks often need more than static slides. Here is how to build a talk in React that can explain systems, states, and interactions more clearly.
---

Many technical talks are trying to explain things that are inherently dynamic:

- state changes
- architecture flows
- user journeys
- step-by-step abstractions
- code evolving over time

Static slides can do some of that. But they often force you into one of two bad outcomes:

- slides that are too dense
- slide sequences that repeat almost the same layout over and over

This is where an interactive, code-first deck starts to make sense.

# What "interactive" should mean

Interactive does not mean gimmicky.

In a good technical presentation, interactivity should help the audience understand:

- what changed
- why it changed
- what part matters right now

That is why the most valuable interaction patterns are usually simple:

- progressive reveal
- step-based diagrams
- embedded UI states
- code that updates focus over time

## Start with the audience's cognitive load

Technical speakers often try to show too much at once.

If a slide contains an architecture diagram, several labels, code, and verbal explanation, the audience has to decide what to pay attention to. That decision costs energy.

A better approach is to control attention deliberately.

Slide-CN includes a `Reveal` component for exactly this reason. It lets you sequence information instead of forcing all of it onto the screen at once.

For example:

```tsx
<Reveal>
  <li>Request enters through the API layer</li>
  <li>Events are queued for processing</li>
  <li>Workers fan out to specialized services</li>
  <li>UI updates when the pipeline completes</li>
</Reveal>
```

## Use one stable layout while the content changes

A technical talk gets easier to follow when the frame stays familiar.

Instead of rebuilding every slide visually, keep the layout stable and update the content inside it:

- same title position
- same code or diagram area
- same explanation panel

That lets the audience focus on the idea, not on re-parsing the whole screen.

Components like `HorizontalSplit`, `VerticalSplit`, and `HeaderWithContent` are useful for this because they create repeatable structure.

## Build one or two high-value interactive moments

Most technical talks benefit from just one or two interactive sequences:

- a diagram that builds in stages
- a demo slide that shows a UI changing state
- a comparison slide with toggled views

That keeps the presentation intentional.

When every slide is active, none of them feels important.

## When this is too much

If you just need a straightforward internal talk with bullet points and a few screenshots, a simpler slide workflow may be better.

Code-first presentations pay off when the talk benefits from composability and state, not when you are solving a purely authoring problem.
