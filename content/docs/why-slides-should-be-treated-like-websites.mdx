---
title: Why Slides Should Be Treated Like Websites
description: Presentations deserve the same system thinking as the rest of your product work. Here is why code-first slides feel fundamentally different from traditional slide software.
---

Most slide tools still assume that presentations are disposable design files.

That works when a deck is a one-off artifact. It breaks down when your team wants the presentation to reflect the same product language, interaction patterns, and visual standards used everywhere else.

The thesis behind Slide-CN is simple:

# Slides should be treated like websites

Not because every talk needs to be flashy. Not because every presentation needs custom code.

Because the best presentations for product teams already behave like product surfaces:

- they use a real design system
- they reuse components
- they share brand tokens
- they can include live UI states, demos, and interactions
- they benefit from version control and iteration

If that sounds more like building a page than decorating a slide, that is the point.

## The problem with treating slides like isolated files

Traditional slide software is optimized for layout editing.

That is useful, but it creates a few recurring problems for developer teams:

### 1. The deck drifts away from the product

Your app uses real spacing tokens, real cards, real button styles, real typography.

Your deck usually does not.

So every presentation becomes a manual imitation of the product instead of an extension of it.

### 2. Reuse is shallow

You can duplicate slides, copy styles, and apply themes.

What you usually cannot do is say:

"Use the same component logic we already trust in our app."

That means each deck starts from visual approximation instead of actual reuse.

### 3. Interactivity is bolted on

Modern product presentations often need more than static images:

- progressive disclosure
- UI state transitions
- architecture diagrams that evolve
- embedded demos
- interactive examples

Those are natural in code. They are awkward in tools built mainly around manual editing.

## What changes when you treat slides like websites

When a presentation is built with React components, a few important things change.

### You design with systems, not screenshots

Instead of re-creating a card by hand, you can render a card.

Instead of matching your product spacing visually, you can use the same spacing tokens.

Instead of inventing a parallel design language for presentations, you can keep one visual system.

### You can compose presentations like interfaces

A deck is no longer a stack of manually designed pages. It becomes a structured UI:

```tsx
<Deck>
  <Slide>
    <TitleSlide>
      <TitleSlide.Heading>Slides should feel like product work</TitleSlide.Heading>
      <TitleSlide.SubHeading>
        Build presentations with the same system your team already uses.
      </TitleSlide.SubHeading>
    </TitleSlide>
  </Slide>

  <Slide>
    <HorizontalSplit
      left={<ProblemPanel />}
      right={<SolutionPanel />}
    />
  </Slide>
</Deck>
```

That is a different mental model.

You are not decorating independent rectangles. You are composing a narrative from reusable pieces.

### Your deck can evolve like the rest of your codebase

When presentations live in code:

- changes are reviewable
- iterations are easier to track
- visuals can stay aligned with app updates
- teams can collaborate without passing around duplicate files

That matters more than people expect, especially once a deck becomes something you return to over time.

## This is not about making every presentation complex

Code-first slides are not automatically better.

If you need to make a quick internal deck for a meeting in twenty minutes, a traditional slide tool may still be the fastest path.

The advantage shows up when the deck needs one or more of these:

- strong visual alignment with a product or brand
- reusable layouts and content patterns
- interaction or live UI behavior
- close collaboration with engineers or designers
- repeatable templates for multiple talks

In other words: when the presentation starts behaving like an actual product asset.

## Why this matters specifically for React teams

React teams already know how to think in components.

They already have:

- a UI layer
- tokens
- patterns
- conventions
- a deployment and iteration workflow

So the real question is not "why use code for slides?"

It is:

## Why switch to a completely separate system for one of the most visible ways you explain your product?

That is the gap Slide-CN is trying to close.

It gives you deck primitives like `Deck`, `Slide`, layout components, reveals, code blocks, and theming, but lets you stay inside the ecosystem you already know.

## When this approach is a great fit

Treat slides like websites when:

- you are presenting a product, platform, or developer tool
- your team already works in React and Tailwind
- consistency matters more than quick drag-and-drop editing
- you want demos and storytelling in the same surface
- you plan to reuse a presentation structure more than once

## When it is not

Slide-CN is probably the wrong tool if:

- the presenter is uncomfortable editing code
- your workflow depends on non-technical collaborators making rapid visual edits in a slide UI
- you only need simple static slides and speed matters more than reuse

That is not a weakness. It is just the tradeoff.

Good tools should be clear about their fit.
